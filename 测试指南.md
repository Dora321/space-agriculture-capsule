# 太空农业种植舱 — 测试指南

## 测试架构总览

```
Level 1 ──→ Level 2 ──→ Level 3 ──→ Level 4
ESP32       PC 端       ESP32        长时间
REPL 单模块  pytest      系统集成     运行验收
```

| 级别 | 环境 | 目的 | 优先级 |
|------|------|------|:------:|
| Level 1 | ESP32 REPL | 逐模块验证硬件通信 | P0 |
| Level 2 | PC (pytest) | Mock 硬件，测试纯逻辑 | P0 |
| Level 3 | ESP32 实机 | 全模块联调，模拟完整循环 | P1 |
| Level 4 | ESP32 长时间 | 72h 稳定性 + 异常注入 | P2 |

**执行顺序**：Level 2（无硬件依赖，随时可跑）→ Level 1（需要 ESP32 在手）→ Level 3 → Level 4。

## 前置条件

### 工具安装

```bash
# Python 环境（需要 Python 3.10+）
py -m pip install mpremote pytest

# 验证安装
py -m pytest --version
py -m mpremote devs
```

### ESP32 连接

1. 用 USB 线将 ESP32 连接电脑，确认 COM 口（Windows 设备管理器查看）
2. 关闭 Thonny 等占用串口的软件
3. 确认 mpremote 能识别设备：

```bash
py -m mpremote devs
# 应输出类似：COM3 (Silicon Labs CP210x)
```

### mpremote 常用命令速查

```bash
# 连接并执行单行代码
py -m mpremote connect COM3 exec "import sensors; sensors.test_all()"

# 上传文件到 ESP32
py -m mpremote connect COM3 cp config.py :

# 进入交互 REPL
py -m mpremote connect COM3

# 软复位（中断正在运行的程序）
py -m mpremote connect COM3 soft-reset

# 查看设备上的文件
py -m mpremote connect COM3 ls
```

> **注意**：每次 `exec` 是独立会话，全局变量不会保留。如果需要连续操作（如先 init 再 test），必须在一条 exec 中完成。如果 ESP32 正在运行 main.py，`exec` 会报 `could not enter raw repl`，需先 `soft-reset`。

## Level 1 — ESP32 REPL 单模块测试

> 将代码烧录到 ESP32 后，通过串口 REPL 逐个验证硬件通信。

### 1.1 传感器测试

```bash
py -m mpremote connect COM3 exec "import sensors; sensors.init(); sensors.test_all()"
```

**检查项：**

| 传感器 | 正常范围 | 异常表现 | 排查方法 |
|--------|---------|---------|---------|
| 土壤湿度 | 0-100% | 返回 `None` | 检查接线：ADC→GPIO34 |
| 土壤湿度 | — | 始终 100% | 传感器未插入土壤/悬空时 ADC 读最大值，属正常 |
| CO2 (MH-Z19B) | 400-2000 ppm | 返回 `None` | 检查 UART：TX→GPIO16, RX→GPIO17，确认 TX/RX 没有反接 |
| CO2 (MH-Z19B) | — | 始终 400 | 需预热 30 秒，`CO2_WARMUP_TIME` 控制 |
| DHT22 | 温度 -40~80°C, 湿度 0-100% | 返回 `(None, None)` | 数据线需 10kΩ 上拉到 3.3V；检查 GPIO4 接线 |
| DHT22 | — | ETIMEDOUT | DHT22 读取间隔需 >= 2 秒，连续读取会超时 |
| 拨码开关 | 返回 8 种植物名之一 | 始终同一值 | 检查 PULL_UP 配置和接地 |

**土壤传感器校准：**

```bash

```

需要准备干土和湿土样品，按提示操作。

### 1.5 AI API 测试

```bash
py -m mpremote connect COM3 exec "import ai_client; ai_client.test_api()"
```

**检查项：**

| 检查点 | 正常表现 | 异常排查 |
|--------|---------|---------|
| 请求发送 | "[AI] Sending request..." | 无输出 → API Key 未配置 |
| 响应成功 | "[Test] Success! Decision: {...}" | 见下方故障排查表 |

**AI API 故障排查：**

| 错误信息 | 原因 | 修复 |
|---------|------|------|
| `Failed to parse JSON` / 请求体截断 | MicroPython `urequests` 对中文字符 Content-Length 计算错误 | 确认代码使用 `.encode('utf-8')` 发送字节 |
| `-116 ETIMEDOUT` | API 响应时间超过 `AI_TIMEOUT` | 增大 `config.AI_TIMEOUT`（当前 20 秒） |
| `finish_reason: length` | 推理 token 消耗了 max_tokens | 增大 `max_tokens`（当前 1024） |
| `TLS memory allocation failed` | TLS 握手时内存不足 | 确保 `plants.json` 已分离；`gc.collect()` 在请求前执行 |
| `API key not configured` | 密钥未设置 | 编辑 `config.py` 填入有效密钥 |

### 2.4 conftest.py Mock 说明

conftest.py 在 pytest 启动前注入以下 Mock，使 MicroPython 代码能在 CPython 上运行：

| Mock 模块 | 替代内容 |
|-----------|---------|
| `machine` | Pin / ADC / UART / I2C / reset / deepsleep / reset_cause |
| `network` | WLAN（STA_IF） |
| `dht` | DHT22 |
| `ssd1306` | SSD1306_I2C |
| `framebuf` | FrameBuffer / MONO_HMSB |
| `urequests` | 空模块（AI 测试走 parse 路径，不发真实请求） |
| `ujson` | 标准库 `json` |
| `gc` | mem_free / mem_alloc / collect |

此外，conftest.py 还做了 `builtins.open` 的 monkey-patch：当 `open('plants.json')` 找不到文件时，自动回退到 `esp32_firmware/plants.json` 路径。

### 2.5 常见测试失败排查

| 失败信息 | 原因 | 修复 |
|---------|------|------|
| `AttributeError: module 'config' has no attribute 'PLANT_DB'` | 测试代码直接引用 `config.PLANT_DB`，但已迁移到 `plants.json` | 改用 `config.get_plant_info()` 或直接从 JSON 加载 |
| `Error loading plants.json: [Errno 2] No such file` | `open('plants.json')` 在 PC 端 CWD 下找不到 | conftest.py 的 `open` 补丁会自动处理；如仍失败检查 `esp32_firmware/plants.json` 是否存在 |
| `NameError: name 'gc' isn't defined` | 缺少 gc mock | 在 conftest.py 中添加 `gc` 模块 Mock |

## Level 3 — 系统集成测试（ESP32 实机）

> 在真实硬件上验证完整工作流。**前置条件**：Level 1 所有模块测试通过。

### 3.1 完整启动流程

```bash
# 先 soft-reset 中断 main.py
py -m mpremote connect COM3 soft-reset

# 然后手动触发初始化
py -m mpremote connect COM3 exec "import main; main.init_system()"
```

**检查清单：**

- [ ] OLED 显示启动画面
- [ ] 传感器初始化无报错
- [ ] 执行器默认全部关闭（继电器高电平）
- [ ] WiFi 连接成功（或显示 "WiFi Failed" 优雅降级）
- [ ] CO2 预热倒计时正常显示
- [ ] LED 绿色常亮

### 3.2 传感器离线降级测试

逐个拔掉传感器，观察系统行为：

| 操作 | 预期行为 | 验证方法 |
|------|---------|---------|
| 拔掉土壤传感器 | LED 红闪 + OLED 显示 "OFFLINE: Soil" + 降级为 0% 触发浇水 | 观察 OLED + 串口日志 |
| 拔掉 CO2 传感器 | LED 红闪 + OLED 显示 "OFFLINE: CO2" + 降级为基线值 420ppm | 观察 OLED + 串口日志 |
| 拔掉 DHT22 | LED 红闪 + 降级为 25°C / 60% | 观察 OLED + 串口日志 |
| 全部拔掉 | 三个传感器同时告警，系统不崩溃 | 确认无硬复位，主循环继续运行 |

> 拔传感器前务必断电操作，带电插拔可能损坏 GPIO。

### 3.4 WiFi 断连恢复测试

1. 正常启动系统，确认 WiFi 连接
2. 关闭路由器 WiFi → 系统应自动切换到本地规则
3. 等待一个读取周期（~5 分钟），确认本地决策正常执行
4. 重新开启 WiFi → 系统应在下一轮循环尝试重连

**验证点：**

```bash

```

### 3.5 完整工作流测试

手动触发一个完整的 "读取→决策→执行" 循环：

```bash
py -m mpremote connect COM3 exec "
import main, time
main.init_system()
time.sleep(2)  # 等 CO2 预热

# 读取传感器
soil = main.sensors.read_soil()
co2 = main.sensors.read_co2()
temp, hum = main.sensors.read_dht22()
print(f'Soil:{soil}% CO2:{co2}ppm T:{temp}C H:{hum}%')

# 本地决策
plant_info = main.config.get_plant_info(main.sensors.read_dip_switch())
decision = main.utils.local_fallback_decision(soil, co2, plant_info, 0, time.time())
print(f'Decision: {decision}')

# 执行决策（会实际运行水泵/风扇，注意安全！）
main.actuators.execute_action(decision['action'], decision['duration_sec'])
print('Action executed')
"
```

> **警告**：此测试会实际触发水泵/风扇。确保水泵已放入水中（干烧会损坏），营养液管路已连接。

## Level 4 — 长时间运行验收

> 验证系统在长时间运行下的稳定性。**前置条件**：Level 3 所有测试通过。

### 4.1 稳定性测试（72 小时）

启动系统后让它自主运行，每隔数小时检查一次状态：

```bash
# 检查运行状态（不会中断主循环，可直接执行）
py -m mpremote connect COM3 exec "
import gc, main
gc.collect()
print(f'Free memory: {gc.mem_free()} bytes')
print(f'Allocated: {gc.mem_alloc()} bytes')
print(f'Error count: {main.state.error_count}')
print(f'Action count: {main.state.action_count}')
import time
uptime = time.time() - main.state.start_time
print(f'Uptime: {uptime // 3600}h {(uptime % 3600) // 60}m')
"
```

**通过标准：**

- [ ] 72h 内无自动重启（看门狗未触发）
- [ ] `gc.mem_free()` 稳定在某个范围，无持续下降趋势（内存泄漏）
- [ ] 传感器读数连续有效，无 `None` 堆积
- [ ] OLED 持续刷新，无花屏/死机
- [ ] `error_count` 未达到 `MAX_ERRORS`

**内存泄漏判断方法**：每小时记录一次 `gc.mem_free()` 值。如果 72h 内下降超过 5KB，说明存在泄漏，需排查。

### 4.2 异常注入清单

| # | 异常场景 | 注入方法 | 预期恢复行为 | 验证方法 |
|---|---------|---------|------------|---------|
| 1 | WiFi 中断 | 关闭路由器 | 切换本地规则，WiFi 恢复后重连 | 观察 OLED 显示变化 |
| 2 | AI API 超时 | 临时改 `AI_TIMEOUT=1`，上传后运行 | 回退本地规则，不阻塞主循环 | 串口日志显示 fallback |
| 3 | 传感器全离线 | 断电后拔掉所有传感器线 | 红色告警 + 降级值运行，不崩溃 | 系统不重启，OLED 显示告警 |
| 4 | 连续错误 > 10 次 | 手动设 `state.error_count = 11` | 看门狗触发 `machine.reset()` | 观察设备自动重启 |
| 5 | 电源闪断 | 拔插 USB | ESP32 自动重启，`main.py` 自动执行 | 系统在 30 秒内恢复正常运行 |
| 6 | 水泵空转保护 | 在土壤干燥时触发浇水但不接水泵 | `PUMP_MAX_RUN_SEC` 限制运行时间 | 水泵在 60 秒内自动停止 |
