### Q6: 你们的菜单系统是怎么工作的？

**回答**：
```
"我们的OLED菜单系统有三级结构：

主菜单 → 植物选择 / 天数设置 / 手动控制 / 系统信息
        ↓
子菜单 → 具体数值显示 / 参数调整

操作用4个按键：红键上翻、黄键下翻、绿键确认、蓝键返回。
蓝键单击进入菜单，蓝键在菜单中返回上一级，
逻辑和手机设置菜单一样，小学生也能用。

植物选择是旋转式选择器，光标固定、列表滚动；
天数设置用红/黄键±1，按住不放会自动加速。
这样设置天数很快，不用一下一下按。

这个设计让我们不用接电脑就能调整参数，
适合太空场景——宇航员戴手套也能按。"
```

**⚠️ 注意**：
- 如果评委要现场看，说："我打开电路板演示"
- 如果电路板没电，说："菜单逻辑我用截图展示"

### Q7: 你们为什么用ADC读按键，而不是用4个GPIO？

**回答**：
```
"这是我们的'模拟键盘'设计——

用电阻梯原理：红键330Ω、黄键680Ω、
绿键1.5KΩ、蓝键3.3KΩ，
4个键并联到同一个ADC引脚。

按不同键，ADC读到不同电压值，
代码里用if-else判断电压区间，就知道按了哪个键。

优势：省了3个GPIO口，
ESP32的GPIO很宝贵，省下来可以接更多传感器。

成本：只加了4个电阻，不到5毛钱。"
```

**⚠️ 注意**：
- 如果评委问"按键冲突怎么办？"，回答："我们设计成单键操作，同时按两个键不处理"
- 如果评委问"ADC精度够吗？"，回答："12位ADC，分辨率4096，4个键的分压差足够大，不会误判"

---

### Q8: 你们的代码模块是怎么分工的？

**回答**：
```
"我们的固件有9个核心模块 + 10个运行时/辅助模块：

核心模块：
config.py —— 引脚定义和阈值参数
telemetry.py —— 传感器数据遥测
sensors.py —— 传感器采集
decision.py —— 双决策引擎（规则+AI）
display.py —— OLED三页显示驱动
buttons.py —— ADC模拟键盘读取
menu.py —— OLED菜单系统（含天数选择）
actuators.py —— 水泵和灯光控制
uart_link.py —— ESP32↔树莓派串口通信协议

运行时/辅助模块：
main.py —— 主循环，串联所有模块
state.py —— 全局状态管理
ai_client.py —— AI代理客户端
utils.py —— 通用工具函数
sensor_runtime.py —— 传感器运行时
boot_runtime.py —— 启动运行时
loop_runtime.py —— 主循环运行时
action_runtime.py —— 动作执行运行时
display_runtime.py —— 显示生命周期管理
status_strip.py —— WS2812状态灯条

总共19个模块、3000多行MicroPython代码，
模块化设计，每个模块负责一件事。"
```

### Q9: 双层架构中Pi在线和离线怎么切换？（新增）

**回答**：
```
"我们的双层架构是ESP32（舱内飞控）
+ 树莓派（载荷计算机），通过UART串口通信。

切换逻辑很简单：
树莓派在线 → ESP32收到种植建议 → 优先采纳Pi的建议
树莓派离线 > 30秒 → ESP32自动回退本地规则引擎

关键是：这个切换对植物完全透明。
不管谁在决策，水泵照常浇水、灯带照常补光，
植物感受不到任何区别。

为什么比赛不接树莓派？
因为我们默认关闭UART模式（UART_ENABLED=False）。
比赛场景WiFi稳定、传感器简单，
ESP32+DeepSeek AI完全够用。
树莓派是为'需要本地图像识别'预留的扩展接口，
接上就能用，不接也不影响核心功能。"
```

**补位**（队员1可以接）：
```
"硬件上UART通信用的是两根杜邦线：
TX（GPIO17）接树莓派RX，RX（GPIO16）接树莓派TX。
波特率115200，数据格式是JSON。
我们的uart_link.py专门处理编解码和超时检测。"
```

## 补位话术：评委问"为什么需要AI决策"时

> 当队员3说到"规则做底线，AI做优化"时，你立刻接技术细节：

### 核心补位（3层技术解释）

```
"我来补充一下技术实现——

第一层，为什么规则写不完：
8种作物×3-5个阶段×4个传感器维度，
组合空间至少96种。
而且温度在阈值边界时，该浇水还是该等？
这种灰色地带，if-else写不出来。

第二层，AI的三个触发条件：
阈值触发——传感器数据越界时请AI判断；
环境剧变——温度突然升高3度以上；
定时复核——每2小时让AI检查一次整体状态。
不是每次循环都调AI，太贵也太慢。

第三层，AI的输出有硬性护栏：
浇水最小间隔180秒、
营养液泵最小间隔300秒、
AI建议必须经过规则引擎安全检查才能执行。

AI是参谋，规则是指挥官。
参谋可以提方案，但拍板的是指挥官。"
```

### 如果评委追问"万一AI的建议和规则冲突怎么办？"

```
"规则赢，永远。

我们的架构是'AI提建议，规则做审批'——
AI输出不是直接执行，而是经过规则引擎安全检查。

比如AI建议'现在浇水'，
但规则引擎发现3分钟前刚浇过——
直接驳回，不执行。

再比如AI建议'开营养液泵'，
但温度传感器显示超过35℃——
规则引擎判定高温浇营养液会烧根，
直接拦截。

代码层面，decision.py里的_should_execute()方法
就是做这个安全检查的。"
```

### 如果评委追问"那10%到底是什么？举个例子？"

```
"比如番茄从苗期进入花期的过渡阶段——
苗期要勤浇水、多氮肥；
花期要控水、加磷钾肥。

但这个'过渡'不是按天算的，是看实际状态。
温度偏高，过渡可能提前；
光照不足，过渡可能延后。

本地规则只能写'第N天切换阶段'，
AI可以根据当天的温度、湿度、光照综合判断：
'今天温度偏高，苗期可能提前结束，可以适当控水了。'

这种判断，if-else写不出来。"
```

### 如果评委追问"既然AI更聪明，为什么不全部用AI？"

```
"三个原因：

第一，网络——太空不是随时在线，断网就废了。
第二，延迟——AI请求一次要1-3秒，
土壤已经干了还等AI回复？来不及。本地规则毫秒级。
第三，成本——AI每次调用消耗API额度和带宽，
太空场景下电力和带宽都是稀缺资源。

所以'规则优先、AI按需'是最优架构。"
```

## 补位话术：评委问"双层架构怎么通信"时（新增）

> 当队员3说到"模块化设计"或队员1说到"树莓派"时，你立刻接技术细节：

### 核心补位（JSON-over-UART）

```
"我来补充通信细节——

ESP32和树莓派之间用UART串口通信，
就是两根线：TX发送、RX接收。

通信协议是我们自己设计的JSON格式：
ESP32每60秒打一个JSON包，包含温度、湿度、
光照、土壤湿度、生长阶段、系统状态。

树莓派接收后处理，返回一个JSON建议包：
建议动作（浇水/补光/施肥/等待）、
置信度、理由。

协议版本号是1，支持4种消息类型：
report（上报数据）、advice（种植建议）、
ping（心跳检测）、pong（心跳回复）。

关键设计：
第一条，如果树莓派30秒没回应，
ESP32自动切回本地规则引擎。
第二条，所有串口通信都是非阻塞的，
不会卡住主循环，不影响传感器采集。"
```
