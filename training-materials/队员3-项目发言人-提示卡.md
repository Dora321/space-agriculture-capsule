## 第一步：育种历史故事（5章，要能复述）

### 第1章：远古时代（公元前8000年）

**核心矛盾**：靠运气，周期极长

**记住这句话**：
```
"原始人采集野生小麦，掉在地上的种子第二年长出来。
他们发现：某些野生植株的籽粒不容易脱落（突变），
于是开始有意识地留种。

但问题来了：完全靠运气，
一个有益性状的出现可能要等几百年。

从野生小麦到驯化小麦，用了8000年。"
```

**关键词记忆**：原始人 → 留种 → 靠运气 → 8000年

---

### 第2章：孟德尔（1865年）

**核心矛盾**：可以计划，但还是要种出来看

**记住这句话**：
```
"1865年，孟德尔用豌豆做实验，发现了遗传规律：
性状是由'遗传因子'（后来叫基因）控制的，
有显性/隐性之分，可以预测杂交后代的性状比例。

这让育种从'靠运气'变成了'有计划'。
但问题还是：要种出来看结果，然后筛选。

周期：杂交 → 种一代 → 看性状 → 再杂交 → 稳定需要5-10年。"
```

**关键词记忆**：孟德尔 → 遗传规律 → 有计划 → 5-10年

---

### 第3章：袁隆平（1973年）

**核心矛盾**：产量大幅提升，但周期还是长

**记住这句话**：
```
"1973年，袁隆平先生研发出杂交水稻，
比常规稻增产20-30%，解决了中国人的吃饭问题。

但杂交水稻的研发周期还是5-10年：
选亲本 → 杂交 → 种F1 → 种F2 → 筛选 → 稳定。

到今天，中国的水稻、玉米、小麦育种，
仍然要8-12年才能出一个新品种。"
```

**关键词记忆**：袁隆平 → 杂交水稻 → 增产20-30% → 还是要8-12年

---

### 第4章：太空育种（1987年至今）

**核心矛盾**：诱变强，但反馈周期被"搭载机会"卡死

**记住这句话**：
```
"1987年，中国开始太空育种实验。
把种子送上太空（返回式卫星、神舟飞船），
利用宇宙射线、微重力、高真空诱变。

太空诱变频率比地面高3-4倍，突变谱也更宽。
中国已经有240多个太空育种品种通过审定，
比如'航芝1号'芝麻，含油量提高5%。

但问题来了：
种子搭上去飞一圈，回到地面还要种5-8代，
才能筛出有价值的突变。

反馈周期被'搭载机会'卡死了——
火箭床位有限，不能随时上去。"
```

**关键词记忆**：1987年 → 太空诱变 → 频率高3-4倍 → 但还要5-8代 → 搭载机会稀缺

---

### 第5章：我们的项目（2035年？）

**核心解决方案**：让种植和筛选发生在太空舱里

**记住这句话**：
```
"我们的解决方案是：
让种植和筛选本身就发生在太空舱里！

这样反馈周期可以砍掉一个量级：
不用等火箭搭载，不用回地面种5-8代，
在轨就能完成初筛。

今天它是一个教育原型，

人类的育种史，就是一部缩短反馈周期的历史。
我们的项目，是这段历史的下一个章节。"
```

**关键词记忆**：让种植发生在太空舱里 → 反馈周期砍一个量级 → 教育原型 → 未来可迁移

## 第十三步：为什么需要AI决策（评委极可能追问）

### 核心金句（必须烂熟）

```
规则能覆盖90%的日常，
但剩下10%的灰色地带，
就是你写1000条if-else也枚举不完的。
AI的价值不是替代规则，而是补上那10%。
```

### 标准回答（约40秒）

```
"为什么需要AI？因为组合太多了。

8种作物，每种3-5个生长阶段，
每个阶段受4个因素影响：温度、湿度、光照、土壤。
4×3×8，至少96种组合——
本地规则能覆盖大部分常见情况，
但总有些'规则没写到'的灰色地带。

比如温度刚好在阈值边界、
土壤湿度下降但还没到浇水线、
植物正好从苗期过渡到生长期——
这种多因素交叉的判断，写规则写不完。

所以我们的设计是：
本地规则管90%的日常决策——快、省电、不依赖网络；
AI管那10%的复杂判断——准、灵活、能处理没见过的情况。

但我们不是无脑信AI。
系统对AI的输出做了硬性约束：
浇水有最小间隔，防止AI连续浇水泡烂根；
执行动作有冷却时间，防止AI频繁切换；
所有AI建议都要经过规则引擎的安全检查才能执行。

一句话：规则做底线，AI做优化。"
```

### 如果评委追问"AI和规则冲突怎么办？"

```
"规则赢，永远。

AI是参谋，规则是指挥官。
参谋可以提方案，但拍板的永远是指挥官。

AI建议'现在浇水'，
但规则引擎发现3分钟前刚浇过——
直接驳回，不执行。"
```

### 如果评委追问"既然AI更聪明为什么不全部用AI？"

```
"三个原因：
网络——断网就废了；
延迟——AI要1-3秒，本地规则毫秒级；
成本——太空里电力和带宽都是稀缺资源。

所以'规则优先、AI按需'是最优架构。"
```

### 如果评委追问"模拟键盘是什么意思？"

```
"模拟键盘是我们用1个ADC引脚

红键330Ω、黄键680Ω、

优势：省了3个GPIO口，
ESP32的GPIO很宝贵，省下来可以接更多传感器。

成本：只加了4个电阻，不到5毛钱。"
```

## 第十五步：双层架构话术（新增——评委问"架构怎么扩展"时）

> 当评委问"你们这个系统能扩展吗？""性能不够怎么办？"时使用。

### 核心金句（必须烂熟）

```
ESP32是舱内飞控，负责生死攸关的决策。
树莓派是载荷计算机，负责锦上添花的优化。
Pi是顾问，ESP32是驾驶员——顾问可以离开，驾驶员不能停。
```

### 标准回答（约40秒）

```
"我们的架构是分层的：

ESP32是'舱内飞控'——
负责传感器采集、执行器控制、基础决策。
这是核心层，断网也能独立工作。

树莓派是'载荷计算机'——
负责更复杂的图像识别和数据分析。
这是扩展层，通过UART串口和ESP32通信。

为什么分两层？
因为ESP32内存只有200KB左右，
要跑图像识别或深度学习模型根本不够。
树莓派相当于外挂了一个'大脑扩展槽'。

关键是：这个扩展是可选的。
接上树莓派，决策更精细；
不接树莓派，ESP32自己也能独立运行。
Pi是增强，不是依赖。

这个设计理念迁移到真实太空场景就是：
核心飞控和载荷实验物理隔离——
飞控不受实验干扰，实验不影响飞控安全。"
```

### 如果评委追问"那比赛现场为什么不接树莓派？"

```
"因为我们默认关闭了UART模式。

比赛场景很简单——
WiFi稳定、传感器只有几个、
不需要图像识别。
ESP32+DeepSeek AI完全够用。

树莓派是为了'本地图像识别'预留的接口——
如果未来要加摄像头、做叶片分析、
跑视觉模型，接上树莓派就行了。

这就是模块化设计：
比赛展示核心能力，扩展功能按需加载。"
```

### 如果评委追问"UART通信可靠吗？不会丢数据吗？"

```
"我们专门写了uart_link.py处理通信层，
有帧边界检测、超时重试、错误恢复。

144个专项测试覆盖了：
消息编解码、超时检测、
半包/粘包处理、异常恢复。

而且关键设计是：
串口通信是非阻塞的，
就算树莓派完全挂了，
ESP32主循环不受任何影响。
30秒超时后自动切回本地规则。"
```
