## 第一步：认识6个零件（3秒指认）

| 零件 | 记住这句话 | 实物特征 |
|------|------------|----------|
| ESP32开发板 | "这是大脑" | 最大的那块板子，有USB口 |
| 土壤湿度传感器 | "插土里的探头" | 两根金属针 |

**训练方法**：
- 把零件摆桌上，教练说名字，学生指认

- 目标：3秒内反应

## 第二步：认识4条连线（指出来）

```
土壤湿度传感器  →  插在ESP32的GPIO34（ADC）
DHT11           →  插在ESP32的GPIO4
继电器-水泵      →  插在ESP32的GPIO5
继电器-营养液泵   →  插在ESP32的GPIO18
OLED显示屏(I2C) →  SDA=GPIO21, SCL=GPIO22
```

**不要求懂为什么，只要求能指出来。**

**训练方法**：
- 指着ESP32上的接口，学生说"这是插土壤的"
- 拿着传感器，学生说"插在GPIO34"

## 第三步：3个固定动作（练到肌肉记忆）

### 动作1：指认零件+翻页演示（配合自我介绍）

```

这是DHT11温湿度传感器，这是继电器模块。

（按黄键）第一页，土壤湿度实时大字；
（按黄键）第二页，生长进度条和施肥建议；
（按黄键）第三页，剩余内存和网络状态。

三个页面把系统状态一目了然地展现出来。"
```

**动作分解**：
1. 拿起ESP32 → "这是大脑"
2. 拿起土壤传感器 → "这是插土里的"

**训练目标**：边说边做，不用思考

### 动作2：拔WiFi演示断网自治（核心演示）

**话术**：
```
"现在我演示断网自治能力。
（拔掉WiFi）

大家看OLED第三页，IP地址区域显示'本地模式'，
系统自动切换到本地规则引擎，继续正常浇水。

模拟火星延迟场景——火星到地球有4-24分钟延迟，
不可能实时遥控。

（插回WiFi）
WiFi恢复后，IP重新显示，系统自动切回AI模式。"
```

**动作分解**：
1. 说"现在我演示断网自治"（手放在USB线旁）
2. 拔掉USB线（或关掉手机热点）
3. 按黄键翻到第三页 → 指IP位置 → "大家看，显示本地模式"
4. 停顿2秒（让评委看清屏幕）
5. 插回USB线
6. 等3秒 → 再指IP → "WiFi恢复，切回AI模式"

**⚠️ 关键**：
- 拔线前要提醒评委"我要拔WiFi了"
- 插回后等3秒，让屏幕切换

## 第五步：可能的Q&A（7个问题）

### Q1: 你们的硬件成本真的只要140元吗？

**回答**：
```
"是的，我们算过：
ESP32开发板 30元，
传感器套件 40元，
继电器和线材 20元，
水泵和灯带 50元，
总共140元。

如果用 Arduino 或者树莓派，成本会翻3-5倍。"
```

---

### Q2: ESP32能稳定运行吗？不会死机吗？

**回答**：
```
"我们有三层保护：
第一，看门狗：如果程序卡死，3秒后自动重启；
第二，内存监控：AI请求前先检查内存，不够就跳过；
第三，故障演练：我们做过断网、传感器失效测试，都能自动恢复。

我们实机跑了30分钟，WiFi零断开，内存稳定。"
```

---

### Q3: 土壤湿度传感器准确吗？

**回答**：
```
"我们做了两点校准：
干燥空气中是0，水中浸泡是100%。

实际使用时，土壤湿度30%以下就浇水，
误差在±5%以内，对于教育原型来说足够了。

如果未来要做科研级应用，可以换工业级传感器。"
```

---

### Q4: 你们的水泵不会烧坏吗？

**回答**：
```
"我们有安全保护：
第一，单次浇水最长10秒，防止干烧；
第二，两次浇水至少间隔5分钟，防止频繁启动；
第三，继电器有指示灯，可以看到水泵是否在运行。

我们测试过，连续运行30分钟没有过热。"
```

---

### Q5: 你们的硬件是自己焊接的吗？

**回答**：
```
"我们用了面包板和杜邦线，没有焊接，
这样方便调试和更换零件。

如果未来要做成产品，会用PCB电路板，
把所有的零件集成到一块板上。"
```

---

### Q6: OLED三页界面分别显示什么？（新增）

**回答**：
```
"我们的OLED屏幕设计了三个信息页，
用红键和黄键手动翻页：

第一页——土壤湿度，用的是2倍大字体，
即使3米外也能看清'65%'这个大数字。
旁边还有作物名称和生长阶段图标（🌱/🌿/🌼）。

第二页——生长进度条，显示当前阶段和进度，
下面标注施肥需求（比如'氮肥'）和水分需求（'正常/偏干'）。
一眼就能知道植物现在需要什么。

第三页——系统资源，显示剩余内存（KB），
MEM状态条（绿色正常/黄色紧张/红色危险），
以及IP地址或'本地模式'。

三页设计的意义是：
太空舱里信息多、屏幕小，
按功能分层展示，宇航员快速定位所需信息，
不用在一个拥挤的界面上找。"
```

**⚠️ 注意**：
- 如果评委想现场看，按黄键翻页演示
- 如果屏幕不亮，用准备的截图代替

---

### Q7: 树莓派是什么角色？你们的双层架构怎么工作？（新增）

**回答**：
```
"在我们的双层架构中——
ESP32是'舱内飞控'，负责传感器采集和执行器控制；
树莓派是'载荷计算机'，负责更复杂的图像和数据分析。

它们之间用UART串口通信（两根线），
ESP32每60秒通过串口发送传感器数据给树莓派，
树莓派处理后返回更精细的种植建议。

但这个架构是可选的——
我们默认关闭UART模式（UART_ENABLED=False），
比赛演示用WiFi+DeepSeek AI模式。

双层架构的意义在于：
ESP32本身内存很小（约200KB可用），
如果要跑图像识别或更复杂模型就不够了。
树莓派相当于给ESP32外挂了一个'大脑扩展槽'，
需要时接上，不需要时ESP32自己也能独立工作。

这就是'模块化设计'——核心功能和扩展功能分离。"
```

## 补位话术：评委问"为什么需要AI决策"时

> 当队员3说到"组合爆炸"或"规则写不完"时，你可以补：

### 补位动作

```
"我用硬件举个例子——
我们8种作物，每种3-5个生长阶段，
4个传感器（温度、湿度、光照、土壤），
4×3×8至少96种组合。

如果用if-else写规则，每加一种作物就要写几十条判断，
还容易漏。

所以本地规则覆盖90%的日常情况——
土壤干了就浇，简单快速；
剩下10%的复杂场景交给AI判断。

但我们加了硬件级护栏：
浇水有最小间隔（防止AI连续浇水泡烂根）、
执行动作有冷却时间（防止频繁切换）、
所有AI建议都要过规则引擎的安全检查。"
```

### 如果评委追问"那拨码开关选作物的逻辑呢？"

```
"拨码开关是3位二进制编码——
DIP1=GPIO13、DIP2=GPIO12、DIP3=GPIO14，
3个开关组合出8种，对应8种作物。

比如拨到110就是十进制6，对应番茄，
ESP32读取后自动从plants.json加载番茄的生长模型。

为什么用拨码不用触屏？
因为太空舱可能戴手套操作，
物理开关比触屏可靠——
这也是我们'离线优先'理念的体现。"
```
