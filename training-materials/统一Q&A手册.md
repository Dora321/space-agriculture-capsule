### Q3: 土壤湿度传感器准确吗？

**标准回答**（队员1）：
```
"我们做了两点校准：
干燥空气中读数是0，水中浸泡读数是100%。

实际使用时，土壤湿度30%以下就浇水，
误差在±5%以内，对于教育原型来说足够了。

如果未来要做科研级应用，
可以换工业级传感器，成本会上升到500-1000元。"
```

---

### Q4: 你们的水泵不会烧坏吗？

**标准回答**（队员1）：
```
"我们有安全保护：

第一，单次浇水最长10秒，防止干烧；
第二，两次浇水至少间隔5分钟，防止频繁启动；
第三，继电器有指示灯，可以看到水泵是否在运行。

我们测试过，连续运行30分钟没有过热。"
```

### Q21: 你们8种作物为什么选这8种？

**标准回答**（队员3）：
```
"我们选了4种叶菜和4种果菜：

叶菜代表'快速实验'——30-50天就能看到完整周期，
适合太空舱短期育种实验。

果菜代表'完整研究'——有花期和果期，
适合研究太空对生殖生长的影响。

两类对比，可以研究一个核心问题：
太空对'营养生长'和'生殖生长'的影响是否不同？"
```

---

### Q22: 你们用的氮磷钾肥料，为什么叶菜只要氮肥？

**标准回答**（队员3）：
```
"因为我们吃的部分不同。

记住口诀：氮长叶、磷开花、钾结果。

叶菜我们吃的是叶子，氮肥就是让叶子长得
又大又绿，所以全程只需要氮肥。

果菜我们吃的是果实，所以要分三步：
苗期氮肥长叶子，花期磷钾促开花，
果期钾肥膨果增甜。

一句话：吃叶子给氮肥，吃果实分三步。"
```

**补位**（队员1可以补）：
```
"我们系统里每种植物都有生长阶段模型，
阶段不同，肥料方案就不同。
比如番茄花期，系统会自动减少浇水、
增加磷钾肥推荐。"
```

---

### Q23: 为什么番茄花期要减少浇水？不怕旱死吗？

**标准回答**（队员3）：
```
"这是农业上的'控水促花'原理。

如果花期水给太多，植物会'贪青徒长'——
一直长叶子不开花。

适当控水，植物感受到'生存压力'，
就会赶紧开花结果，把基因传下去。

我们的系统不是不浇，是'不那么干就不浇'。
番茄花期阈值从35%提高到40%，
只有真的快干了才浇。"
```

---

### Q25: 太空种菜和地面种菜有什么不同？

**标准回答**（队员3）：
```
"最大的不同有三个：

第一，微重力——根不知道往哪长，
水也不会自然往下流。
太空种植用'毛细管基质'代替土壤。

第二，光周期——空间站每90分钟绕地球一圈，
太空一天有16次日出日落，
植物会'晕'，必须人工控制光照。

第三，水循环——太空里水非常珍贵，
宇航员连汗都要回收。
太空种菜的水必须是闭环的。

我们的项目模拟了第三点：
用土壤湿度传感器精确控制浇水，避免浪费。"
```

## 第九部分：双层架构与扩展（队员1/2主答，队员3补位）

> ⚠️ 新增部分，对应最新代码（uart_link.py + 树莓派双层架构）

### Q43: 你们说的双层架构是什么意思？ESP32和树莓派怎么分工？

**标准回答**（队员1主答）：
```
"我们把系统分成两层：

ESP32是'舱内飞控'——
负责传感器采集、执行器控制、基础决策。
这是核心层，所有'生死攸关'的操作都在这一层。
即使没有任何外部设备，ESP32也能独立运行。

树莓派是'载荷计算机'——
负责更复杂的计算，比如图像识别、数据分析。
它通过UART串口和ESP32通信，
ESP32每60秒发送一次传感器数据，
树莓派处理后返回更精细的种植建议。

关键设计理念：
两层物理隔离，互不干扰。
就算树莓派完全死机，
ESP32的主循环不受任何影响，
30秒超时后自动切回本地规则引擎。"
```

**补位**（队员2可以接）：
```
"从软件角度补充——

两层之间的通信协议是我们自己设计的JSON-over-UART。
4种消息类型：report（上报数据）、advice（种植建议）、
ping（心跳检测）、pong（心跳回复）。

关键约束：
所有串口通信都是非阻塞的，不会卡住主循环。
树莓派在线时，优先采纳Pi的建议；
树莓派离线超过30秒，ESP32自动回退本地决策。"
```

---

### Q44: UART通信可靠吗？太空环境电磁干扰怎么办？

**标准回答**（队员2主答）：
```
"我们做了多层可靠性保障：

第一层，协议层——每帧数据有明确的开始/结束标记，
如果收到半包或乱码，直接丢弃等下一帧。

第二层，超时检测——如果30秒内没收到树莓派的完整回复，
ESP32判定树莓派离线，切回本地模式。

第三层，故障恢复——树莓派重新上线后，
ESP32收到第一个ping的pong回复后，
自动恢复数据上报和Pi建议接收。

第四层，测试保障——我们写了144个UART专项测试，
覆盖消息编解码、超时检测、
半包/粘包处理、异常恢复、上线/离线切换。

所有测试全部通过。"
```

### Q45: 树莓派是必须的吗？没有树莓派系统能跑吗？

**标准回答**（队员3主答）：
```
"树莓派不是必须的，它是一个'增强模块'。

没有树莓派：ESP32自己完成采集→决策→执行，
通过WiFi调用DeepSeek AI提供智能建议。
这就是比赛现场的模式。

有树莓派：ESP32把传感器数据通过串口发过去，
树莓派可以用本地模型（不需要网络）做更精细的分析，
比如图像识别叶片健康状况。

打个比方——
树莓派就像汽车的涡轮增压：
没有涡轮，车照样能开（自然吸气）；
有涡轮，高速时才爆发更强动力。
但日常通勤（比赛场景），自然吸气完全够用。

这个设计理念很重要：
核心功能和扩展功能分离，
扩展模块的存在不影响核心模块的独立性和稳定性。"

连UART驱动都不会加载，零开销。
设为True，自动初始化UART2，开始和树莓派通信。
```

### Q46: ESP32同时跑WiFi+UART+OLED，内存够吗？你们怎么解决的？

**标准回答**（队员2主答）：
```
"这个问题问到了我们在开发中遇到的一个实际困难。

ESP32的可用堆内存约200KB。
同时初始化WiFi、UART2和OLED/I2C三者时，
堆内存压力非常大，
会出现UART驱动malloc失败或OLED底层崩溃。

我们的解决方案分两步：

第一步，模式隔离——
当UART模式启用时（UART_ENABLED=True），
自动跳过WiFi初始化（UART_SKIP_WIFI=True）。
因为树莓派本身就提供网络，
ESP32不需要再连WiFi。

第二步，初始化顺序——
先初始化UART2，再初始化OLED。
这个顺序确保了内存分配优先级：
通信层（UART）优先于显示层（OLED）。

实机测试验证了这个方案：
OLED正常显示三页内容，
UART通信稳定，
系统内存余量充足。"
```

**补位**（队员1可以接）：
```
"这是我们在实机调试中发现的——
ESP32的资源非常有限，
模块之间会互相抢占内存。

我们的解决思路是'做减法'：
不是所有功能都同时开启，
而是根据场景按需加载。

比赛WiFi模式 → 不加载UART驱动
UART扩展模式 → 跳过WiFi初始化

就像手机的低功耗模式——
关掉不需要的功能，把资源留给核心任务。"
```

**补位**（队员3可以补）：
```
"这个问题的解决过程也体现了
我们项目的工程思维——

发现问题（内存冲突）→
定位根因（用mem_profile.py做内存分析）→
设计方案（模式隔离+初始化顺序）→
测试验证（144个UART测试全部通过）。

这就是从'能跑'到'稳定'的工程化过程。"
```
