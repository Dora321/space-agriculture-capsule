# 仓库架构评估

更新时间：2026-05-21

## 总体结论

当前仓库已经完成从“单文件主控 + 展示材料堆叠”到“分层固件 + 自动化测试 + 交付物索引”的整理。固件核心路径已经按启动、主循环、显示、采集、决策、执行和状态拆分，后续修改可以围绕单一职责模块进行。

项目现阶段适合作为竞赛展示和教学演示交付；如果要继续向长期运行设备演进，下一阶段重点应转向实机验收、运行日志、配置治理和硬件在环测试。

## 当前结构

```text
.
├── esp32_firmware/          # ESP32 MicroPython 固件
├── tests/                   # PC 端自动化测试与 MicroPython mock
├── tools/                   # Dashboard、AI 代理、生成脚本
├── deliverables/            # 大屏、KT 板、展示和验收材料
├── README.md                # 项目总入口
├── 测试指南.md              # 测试说明
└── 智能种植舱控制器选型报告.md
```

## 固件分层

| 模块 | 职责 |
| --- | --- |
| `main.py` | 入口、兼容包装、模块接线 |
| `state.py` | 主循环共享运行状态 |
| `boot_runtime.py` | 启动初始化流程 |
| `loop_runtime.py` | 采样、决策、执行、上报、重连的主调度 |
| `display_runtime.py` | OLED 生命周期和分页刷新 |
| `sensor_runtime.py` | 传感器读取编排、离线降级、生长统计 |
| `decision.py` | 本地规则、AI 请求门控、云端决策编排 |
| `action_runtime.py` | 动作执行、安全检查、动作计数 |
| `sensors.py` | 传感器底层读取 |
| `actuators.py` | 执行器底层控制 |
| `telemetry.py` | Dashboard 上报 |
| `ai_client.py` | AI 请求和响应解析 |
| `display.py` | OLED 页面绘制 |

## 已完成的治理

- `config.py` 已从版本控制中排除，保留 `config.py.example` 作为模板。
- 诊断脚本已移动到 `esp32_firmware/diagnostics/`。
- 固件核心运行职责已拆分为多个 `*_runtime.py` 模块。
- 自动化测试覆盖配置、本地规则、AI 解析、遥测、主循环边界和文档质量。
- 交付物目录已有索引，包含大屏指南、KT 板材料和实机验收清单。
- `.gitignore` 已排除 pycache、pytest cache、IDE 文件、本地密钥和预览截图。

## 主要风险

1. **实机长期运行证据不足**  
   PC 端测试能证明逻辑路径，但不能证明水泵、电源、DHT、WiFi 和 OLED 在 ESP32 上长期稳定。

2. **`display.py` 仍偏大**  
   OLED 绘制逻辑集中在单文件，后续若继续增加页面或中文适配，建议再拆页面绘制和文本格式化。

3. **`ai_client.py` 仍承担多种职责**  
   Prompt 构造、HTTP 请求、JSON 解析、错误处理在一个模块内，后续可拆为 prompt、transport、parser。

4. **配置项继续增长会变难维护**  
   当前单个 `config.py.example` 还能接受；如果继续增加演示参数、硬件参数和网络参数，建议按章节重排或拆配置域。

5. **硬件动作仍是阻塞式**  
   MicroPython 单线程下水泵动作期间主循环会阻塞，这是已知限制。当前通过最大运行时长控制风险。

## 推荐下一步

| 优先级 | 事项 | 目标 |
| --- | --- | --- |
| 高 | 按 `实机验收清单.md` 跑一次完整硬件验收 | 获得真实设备稳定性证据 |
| 高 | 记录一次 30 分钟运行日志 | 验证 WiFi、内存、遥测和动作安全 |
| 中 | 给 `sensor_runtime.py` 和 `action_runtime.py` 增加直接单元测试 | 减少对 `main.py` 包装测试的依赖 |
| 中 | 梳理 `config.py.example` 分区注释 | 降低现场配置出错概率 |
| 低 | 拆分 `display.py` 页面绘制 | 为后续 UI 扩展留空间 |
| 低 | 拆分 `ai_client.py` | 提升 AI 请求链路可测试性 |

## 当前判断

仓库已经达到“可展示、可维护、可继续迭代”的状态。接下来不建议继续大规模重构，优先用实机验收和小范围测试补强来换取稳定性。
