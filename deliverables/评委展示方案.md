## 二·B 3分钟完整介绍话术（备用菜园版）

### 🎯 第一段：问题与灵感（30秒）

> "太空探索中，宇航员需要新鲜蔬菜补充营养，但太空环境下没人能天天浇水。我们的灵感来自一个问题：**能不能让AI当植物学家，7×24小时照顾太空舱里的蔬菜？** 这就是太空农业智能种植舱的出发点。"

### 🔧 第二段：系统架构（60秒）

> "我们的系统分**三层架构**：
>
> **感知层**——4类传感器：土壤湿度（电容式ADC）、环境光照（光敏ADC）、温湿度（DHT11）、作物选择（3位拨码开关，支持8种作物一键切换）。
>
> **决策层**——双引擎：优先调用**云端AI**（DeepSeek大模型），根据植物种类、生长天数、当前环境综合判断该浇水还是施肥；网络断开时自动切换**本地规则引擎**，确保离线也能正常养护。AI会考虑生长阶段——苗期少浇水、花期减水补磷钾、果期大量水钾肥。
>
> **执行层**——3个继电器控制：水泵、营养液泵、风扇，低电平触发，带安全上限保护（单次最长60秒）。
>
> 整个系统跑在 **ESP32 + MicroPython** 上，代码完全自主编写。"

### 🌱 第三段：核心亮点（60秒）

> "我们项目有**5大亮点**：
>
> **1. AI + 规则双决策引擎**：云端AI理解植物需求，本地规则保证离线可用，两者无缝切换。
>
> **2. 14种作物参数库**：`plants.json` 内置 14 种作物的独立生长阶段表、浇水阈值、施肥配方、光照需求；3 位拨码开关现场切换 8 种核心作物，系统自动适配养护策略。
>
> **3. 生长阶段感知**：系统根据种植天数自动判断当前是苗期、营养生长期、花期还是果期，不同阶段给出不同的水肥策略。
>
> **4. 实时数据大屏**：Web仪表盘展示SVG圆弧仪表、趋势曲线、植物生长动画、决策流水线，支持1920×1080大屏展示。
>
> **5. 全链路安全机制**：看门狗重启、每小时动作上限、最小动作间隔防抖、传感器离线检测降级——确保系统长期稳定运行。"

### 🎬 第四段：现场演示引导（30秒）

> "现在请看现场演示——屏幕上可以看到种植舱的实时数据：土壤湿度35%、光照72%、温度24.5℃……系统刚刚自动判断土壤偏干，执行了一次8秒浇水，决策理由是'soil dry'。各位也可以看到OLED小屏上的三页轮播：传感器页、生长状态页、系统状态页。我们随时可以切换拨码开关，让系统从养护生菜切换到养护番茄，它会立即调整所有阈值和策略。"

## 三、评委可能的追问 + 应答

| 评委可能问 | 建议回答 |
|---|---|
| **AI决策和本地规则有什么区别？** | AI能理解更复杂的场景，比如'光照不足导致蒸发减少，所以即使土壤偏干也不急着浇水'；本地规则是硬阈值判断——土壤低于30%就浇水。AI更聪明但需要网络，规则更可靠但不够灵活，两者互补。 |
| **为什么选ESP32而不是Arduino？** | ESP32自带WiFi和蓝牙，能直接连云端AI；运算能力比传统Arduino强，能跑MicroPython；价格只要二三十块，性价比极高。 |
| **为什么用MicroPython不用C++？** | MicroPython开发速度快，适合快速原型；ESP32内存足够运行Python解释器；我们的AI客户端需要JSON序列化，Python天然支持；C++更适合对实时性要求极高的场景，我们的控制周期是60秒，Python完全够用。 |
| **断网了怎么办？** | 自动切换本地规则引擎，土壤干燥就浇水，营养液到期就施肥，完全离线也能正常运行。 |
| **数据怎么传到大屏？** | ESP32通过WiFi把状态JSON POST到云端服务器，大屏网页轮询获取数据，实时刷新。 |
| **14种作物的数据从哪来？** | 我们查阅了农业种植手册，整理了每种作物的生长阶段、浇水阈值、施肥配方、光照需求，写成 `plants.json` 数据库。3 位拨码现场切换 8 种核心，另外 6 种（豆角、西葫芦、萝卜、大蒜、葱、生姜）可通过代码修改植物名直接调用。 |
| **成本多少？** | ESP32开发板30元 + 传感器约20元 + 继电器模块15元 + 水泵营养液泵约30元 + OLED屏幕10元 + 其他配件约20元，总计约125元。 |
| **这个系统实际种过菜吗？** | 有实际种植测试，自 `PLANTING_DATE`（详见 `config.py`）起在工位连续运行至今，累计采样次数、浇水/施肥次数请现场指向 Web 大屏右上角的 `采样次数 / 动作次数` 实时计数（这两个数由设备上报，不依赖话术口播）。<br>**答辩前 24h 务必现场操作一次**：打开大屏 → 记录当前 `read_count` 与 `action_count` → 把数字写入下面这行 → 演讲时直接说出。<br>`【现场填写】运行 ____ 天 / 采样 ____ 次 / 动作 ____ 次（_____ 月 _____ 日 _____ 时核对）` |

### 三 · A 套（重立意版）追加追问 — 育种科学相关

> 用 A 套立意时，这 5 题大概率会被大学教授评委问到，必须提前练熟。

| 评委可能问 | 建议回答 |
|---|---|
| **你了解中国的航天育种吗？** | "了解。1987 年中国第一次用返回式卫星搭载水稻和青椒种子，是世界上少数早期开展太空育种的国家之一。2006 年的'实践八号'是专门的育种卫星，搭载了 9 类 152 种农作物种子。到今天，中国已经培育出 240 多个航天育种新品种，'航聚香丝苗'水稻和'鲁原 502'小麦是已经大规模推广的代表。航天育种是我们国家真正领先的领域之一。" |
| **宇宙射线为什么能诱变？** | "宇宙射线里有高能质子、重离子和 γ 射线，能量比地面 X 光高几个数量级。它打到 DNA 链上能直接造成断裂、点突变、片段缺失或者易位。同时微重力和高真空让细胞的 DNA 修复机制运行异常——平时能修好的损伤这时候修不全，反而保留下来变成可遗传的突变。结果就是突变频率比地面诱变高 3 到 4 倍，突变谱也更宽。" |
| **你这个项目和真正的航天育种是什么关系？** | "直接关系是：**地面原型 → 未来的在轨平台**。<br>真正的在轨育种还要解决很多额外问题——微重力下水分子聚集、根系方向紊乱、宇航员级安全标准、辐射防护。我们这套地面原型没有解决这些。<br>但我们解决的是**控制架构**：多品种平行种植、高密度数据采集、AI 实时分析、远程遥测、自治容错——这套架构在地面跑通之后，未来上轨道时不用从零开始，少走很多弯路。" |
| **现有航天育种已经成功了，你做'在轨育种'有什么必要性？** | "现有方案是'**太空诱变 + 地面验证**'，种子飞一趟回来还要在地面再种 5 到 8 代才能筛出来——反馈周期被'搭载机会'和'地面田间试种'卡住了。<br>'**在轨育种**'是'**太空诱变 + 太空验证**'——节省地面田间试种时间；而且在真实的太空环境里选育的品种，本来就是'天生适合太空'的；同时数据可以实时回传地面，让地面 AI 接力筛选，形成天地协同。<br>这条路现在还没人完整做出来——这是我十年后想做的事。" |
| **你的项目最难的部分是什么？/ 你从这个项目学到什么？** | （**这两题是软问题，是 5 年级孩子的机会题，要给"自己的话"**）<br>**最难的部分**："对我来说最难的不是写代码——代码很多是和老师一起做的。最难的是想清楚'**万一**'：万一断网怎么办、万一传感器坏了怎么办、万一温度突然飙到 40 度怎么办。平时系统看起来很简单，但要把所有'万一'都想到，比想'平时'难得多。"<br>**学到了什么**："学到了一个事——**人类一万年来都在做同一件事**，就是缩短育种的反馈速度。这件事我以前不知道。我做完这个项目之后，看新闻里讲'神舟飞船带种子上太空'就不再是看热闹，而是知道它在解决一个具体的问题。"

## 六、现场演示 Checklist

### 硬件准备
- [ ] ESP32开发板 + USB供电
- [ ] 传感器接线正常（土壤/光照/DHT/拨码）
- [ ] 继电器 + 水泵/营养液泵接线正常
- [ ] OLED显示正常
- [ ] WiFi连接正常
- [ ] 状态LED正常（绿=正常/黄=执行中/红=故障）

### 软件准备
- [ ] MicroPython固件已烧录
- [ ] config.py中WiFi密码正确
- [ ] AI API Key有效
- [ ] Web大屏浏览器已全屏打开
- [ ] Demo模式已关闭（展示真实数据更好）

### 演示脚本
1. **开场**：指着OLED说"这是种植舱的实时状态"
2. **拨码切换**：拨动开关，OLED和大屏同步切换作物类型
3. **触发浇水**：用手捏住土壤传感器（模拟干燥），观察AI决策→水泵动作→OLED和大屏更新
4. **断网测试**：断开WiFi，展示本地规则接管
5. **大屏展示**：引导评委看大屏上的趋势曲线和植物生长动画

## 七、加分项建议

1. **太空主题外观**：给种植舱外壳贴上太空舱风格的装饰（银色铝箔纸、LED灯带）

3. **延时视频**：用手机拍摄7天延时视频，展示植物在系统照料下的生长过程
4. **数据报告**：打印一份运行7天的数据统计报告（浇水次数、施肥次数、土壤湿度曲线）
