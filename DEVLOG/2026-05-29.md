## 完成事项

### #24 WiFi 连接稳定性

**根因**：ESP32 WiFi 驱动初始化需要约 130KB 连续堆内存（10 个 rx 缓冲区 × 1600B）。
`utils.py` 顶层 `import status_strip`（11KB），`boot_runtime` / `sensor_runtime` 顶层 `import utils`，
这两条链在 `main.py` 启动时就被拉入，导致 WiFi init 前可用内存只剩 129KB 且碎片严重。
实际可用连续块不足，报 `Expected to init 10 rx buffer, actual is 3` → OOM。

**修复**：

`main.py`:
- 顶层 import 精简为 `config / wifi_client / display_runtime / state`（轻量集合）
- `boot_runtime / sensor_runtime / action_runtime / decision / loop_runtime` 全部改为首次调用时懒加载
- `init_system()` 在 `import boot_runtime` 之前先执行 `wifi_client.connect()`（此时只有轻量模块在堆）

`boot_runtime.py`:
- 新增 `wifi_already_connected` 参数，跳过重复 WiFi 连接
- `import actuators / import utils` 移到 `init_system()` 内、WiFi 连接完成之后

效果：WiFi init 前可用内存从 **129KB → 149KB**（+20KB），消除 OOM。

`wifi_client.py`:
- `connect()` 加入 `active(False)` → `sleep(1)` → `active(True)` → `sleep(1)` 序列
- 修复运行期掉线后重连触发 `RuntimeError: Wifi Unknown Error 0x0102`（WiFi 驱动状态机卡死）

### #25 按键响应改善

**根因 1（最严重）**：`loop_runtime.py` 主循环末尾 `time.sleep(1)`，`check_menu()` 每秒只轮询一次，
按键最差要等 1 秒才被检测到，长按进菜单实际等待可达 2 秒。

**修复**：`time.sleep(1)` → 100ms 子循环轮询 9 次，按键检测频率 1Hz → **10Hz**。

**根因 2**：`DEFAULT_THRESHOLDS` 中 DOWN 按键上限 3100 < 实测最大值 3166，
ADC 读到 3101–3166 时被归入 UP 范围，导致 DOWN 键偶发误触发 UP 或漏键。

**修复**：DOWN 阈值上限 3100 → **3200**，UP 下限 3101 → 3201（与实测 UP 最小值 3242 仍有 41 单位裕量）。

**其他**：ADC 均值采样次数 4 → 8，降低噪声对阈值判断的影响。

### #26 菜单黑屏修复

**根因**：`_check_menu()` 原来的序列是 `release_display()` → `_init_display()` → `run_main_menu()`。
`release_display()` 把 `display` 模块从 `sys.modules` 删除并置 `_oled = None`，
`_init_display()` 随后创建全新模块对象，但 `Menu._display` 存的是初始化时的旧引用（`_oled = None`）。
所有绘图调用命中 `_check_init()` 提前 return → 菜单全程黑屏，直到退出后 `refresh_display()` 用新模块渲染才亮。

**修复**：删除 `_check_menu()` 中无意义的 `release_display()` + `_init_display()` 对，
改为进入菜单前执行一次 `_menu._display = display_runtime.display()`，
`display_runtime.display()` 是惰性的：有效直接返回，被 AI 请求释放过则先重初始化再返回。
覆盖「AI 请求后 display 被重建」场景。

### #27 菜单交互统一（无长按）

**问题梳理**（原有 4 处不同时长的长按）：

| 场景 | 原来 | 修复后 |
|------|------|--------|
| 主界面进菜单 | 蓝键长按 1000ms | 蓝键单击 |
| 菜单内返回 | 蓝键长按 1500ms | 蓝键单击 |
| 植物选择取消 | 蓝键长按 2000ms | 蓝键单击 |
| 系统信息返回 | 蓝键长按 1500ms | 蓝键/绿键单击 |

**统一规则**：红/黄 = 上下导航；绿 = 确认；蓝 = 返回/退出（全局，无例外）

`buttons.py`:
- 新增 `back_pressed()` —— 蓝键单击事件，语义与 `pressed()`（绿键）完全对称
- `pressed()` 去除 fallback `_event()` 调用，简化状态机

`menu.py`:
- 删除所有 `long_pressed()` 调用（4 处）
- `check_menu_trigger()` 改用 `back_pressed()`
- 删除主菜单"Back"条目（蓝键直接退出已覆盖）
- `run_plant_selection` 循环改用取模，支持首尾循环

`display.py`:
- 底部提示统一改为 `"<>nav  G:ok  B:bk"`（全部界面一致）
- `show_system_info()` `"Long press:Back"` → `"Blue: Back"`

`display_runtime.py`:
- 新增 `advance_page(delta)` —— 主界面红/黄键翻页入口

## 遇到的问题

1. **WiFi OOM 时好时坏**：Ctrl-D soft-reset 后可用内存分布与 power-on reset 不同（碎片程度不同），
   导致相同总可用内存下行为不一致。通过"WiFi 先于重模块加载"而非单纯增加 gc.collect() 次数来根治。

2. **main.py 被卡在菜单等待按键**，无法通过 mpremote cp 传文件（"could not enter raw repl"）：
   先 soft-reset 中断菜单循环，再上传。

3. **ESP32 串口输出捕获**：mpremote exec 打断 main.py 后模块全局变量清空，
   用 `serial` 直连 + Ctrl-D 的方式才能正确捕获 boot 日志。
