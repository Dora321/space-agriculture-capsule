## 完成事项

### #24 WiFi 连接稳定性

**根因**：ESP32 WiFi 驱动初始化需要约 130KB 连续堆内存（10 个 rx 缓冲区 × 1600B）。
`utils.py` 顶层 `import status_strip`（11KB），`boot_runtime` / `sensor_runtime` 顶层 `import utils`，
这两条链在 `main.py` 启动时就被拉入，导致 WiFi init 前可用内存只剩 129KB 且碎片严重。
实际可用连续块不足，报 `Expected to init 10 rx buffer, actual is 3` → OOM。

**修复**：

`main.py`:
- 顶层 import 精简为 `config / wifi_client / display_runtime / state`（轻量集合）
- `boot_runtime / sensor_runtime / action_runtime / decision / loop_runtime` 全部改为首次调用时懒加载
- `init_system()` 在 `import boot_runtime` 之前先执行 `wifi_client.connect()`（此时只有轻量模块在堆）

`boot_runtime.py`:
- 新增 `wifi_already_connected` 参数，跳过重复 WiFi 连接
- `import actuators / import utils` 移到 `init_system()` 内、WiFi 连接完成之后

效果：WiFi init 前可用内存从 **129KB → 149KB**（+20KB），消除 OOM。

`wifi_client.py`:
- `connect()` 加入 `active(False)` → `sleep(1)` → `active(True)` → `sleep(1)` 序列
- 修复运行期掉线后重连触发 `RuntimeError: Wifi Unknown Error 0x0102`（WiFi 驱动状态机卡死）

### #25 按键响应改善

**根因 1（最严重）**：`loop_runtime.py` 主循环末尾 `time.sleep(1)`，`check_menu()` 每秒只轮询一次，
按键最差要等 1 秒才被检测到，长按进菜单实际等待可达 2 秒。

**修复**：`time.sleep(1)` → 100ms 子循环轮询 9 次，按键检测频率 1Hz → **10Hz**。

**根因 2**：`DEFAULT_THRESHOLDS` 中 DOWN 按键上限 3100 < 实测最大值 3166，
ADC 读到 3101–3166 时被归入 UP 范围，导致 DOWN 键偶发误触发 UP 或漏键。

**修复**：DOWN 阈值上限 3100 → **3200**，UP 下限 3101 → 3201（与实测 UP 最小值 3242 仍有 41 单位裕量）。

**其他**：ADC 均值采样次数 4 → 8，降低噪声对阈值判断的影响。

### #26 菜单黑屏修复

**根因**：`_check_menu()` 原来的序列是 `release_display()` → `_init_display()` → `run_main_menu()`。
`release_display()` 把 `display` 模块从 `sys.modules` 删除并置 `_oled = None`，
`_init_display()` 随后创建全新模块对象，但 `Menu._display` 存的是初始化时的旧引用（`_oled = None`）。
所有绘图调用命中 `_check_init()` 提前 return → 菜单全程黑屏，直到退出后 `refresh_display()` 用新模块渲染才亮。

**修复**：删除 `_check_menu()` 中无意义的 `release_display()` + `_init_display()` 对，
改为进入菜单前执行一次 `_menu._display = display_runtime.display()`，
`display_runtime.display()` 是惰性的：有效直接返回，被 AI 请求释放过则先重初始化再返回。
覆盖「AI 请求后 display 被重建」场景。

## 遇到的问题

1. **WiFi OOM 时好时坏**：Ctrl-D soft-reset 后可用内存分布与 power-on reset 不同（碎片程度不同），
   导致相同总可用内存下行为不一致。通过"WiFi 先于重模块加载"而非单纯增加 gc.collect() 次数来根治。

2. **main.py 被卡在菜单等待按键**，无法通过 mpremote cp 传文件（"could not enter raw repl"）：
   先 soft-reset 中断菜单循环，再上传。

3. **ESP32 串口输出捕获**：mpremote exec 打断 main.py 后模块全局变量清空，
   用 `serial` 直连 + Ctrl-D 的方式才能正确捕获 boot 日志。

### #28 开机前闪第 1 页修复

**根因**：`init_system()` 调用 `boot_runtime.init_system()` 时传入了 `refresh_display` 回调，
boot_runtime 在内部渲染了一次仪表盘，此时蔬菜/天数尚未选择，导致开机时先闪一下 Page 1（传感器页）
再进入植物选择界面。

**修复**：`main.py` 向 `boot_runtime.init_system()` 传入 `refresh_display=None`，
完全禁止 boot_runtime 在选择环节前渲染仪表盘。
仪表盘首次渲染移到 `_select_plant()` + `_select_day()` 全部完成之后：

```python
_select_plant()
_select_day()
_refresh_display(force=True, reset_page=True)  # 首次渲染，选完才出现
```

### #30 手动操作后卡在第 1 页（按键/翻页全失响）

**根因 1（页面重置）**：传感器采样块末尾调用 `refresh_display(force=True, reset_page=True)`，
每 60 秒强制把页码跳回第 0 页，看起来"卡在第 1 页"。

**修复**：传感器采样块改为 `refresh_display(force=True)`（无 `reset_page=True`），
仅在菜单退出时使用 `reset_page=True`。

**根因 2（立刻触发 AI）**：手动操作（`run_water_pump` 阻塞 30s）结束后回到主循环，
`last_read` 是旧时间戳 → `now - last_read ≥ interval` 立即成立 → 触发传感器采样 + AI 请求
再冻结一段时间，期间 OLED 静止在第 1 页，按键无响应。

**修复**：菜单退出时（不论内层/外层轮询）重置 `last_read = int(time.time())`，
延迟 `READ_INTERVAL` 秒后才进行下一次采样，给系统喘息时间。

### #32 WiFi 重连冻结根治（77s → 9s）

**根因**：WiFi 掉线时 `loop_runtime.py` 调用 `wifi_client.smart_connect()`，
该函数最多重试 3 次，每次最长 `0.6s（active 复位）+ 20s（超时）+ 5s（等待）= ~25.6s`，
3 次合计约 **77 秒** 主循环完全冻结——OLED 停在当前页，所有按键无响应。

**修复 1** — `wifi_client.py`：
`active(False)→active(True)` 复位间隔从 `time.sleep(1)` 改为 `time.sleep_ms(300)`（每次节省 700ms）

**修复 2** — `loop_runtime.py`：
单次连接替代多次重试：

```python
# 之前（最坏 77s）
wifi_client.smart_connect()

# 之后（最坏 ~9s；失败则下次循环 60s 后再试）
wifi_client.connect(timeout=8)
```

| | 修复前 | 修复后 |
|--|------|--------|
| active 复位延迟 | 1s + 1s = 2s | 300ms + 300ms = 0.6s |
| 重连策略 | smart_connect 3 次×25s | connect(timeout=8) 单次 |
| 最长冻结时长 | ~77s | ~9s |

## 现场调试 #33 — Dashboard 无数据 / OLED 显示 "no connection" 根因排查

> ⚠️ **本节关于 brownout 的结论已被 #34 推翻**：当时把"12V 水泵吸合 brownout"当成根因，
> 但后续发现测试时 **12V 电源根本没打开、水泵从未真正运转**——brownout 属误判。据此加的
> BOD 禁用、水泵脉冲分段、action 后 sleep 均已全部回退。以下保留作排查过程记录，结论以 #34 为准。

### 现象

用户反馈：
1. 网页大屏 `43.156.68.157:8790` 始终没有新数据上来
2. OLED 第 3 页 IP 行常显示 `no connection`
3. 看起来"一直在演示模式"（实际未启用 DEMO_MODE，是显示卡在开机初值）

### 分层排查与发现

#### 第 1 层：WiFi 关联本身是好的

- 设备 scan 看到 `machine_maker` RSSI **-50 dBm**（极强）
- 干净环境下手动 `wlan.connect()` 2 秒内拿到 IP `192.168.125.113`，status=1010 (GOT_IP)
- 从 ESP32 直接 `POST /api/state` 到 dashboard 返回 `HTTP 200 OK / {"ok": true}` —— **网络通道完全正常**
- 单独调用 `telemetry.send_state()` 也返回 True

→ 排除"路由器问题"、"dashboard 服务器问题"、"telemetry 代码问题"。

#### 第 2 层：跑完整 `main.py` 后第一次水泵动作就 brownout reset

启动日志：
```
[Water Pump] Starting for 8s
 BOD: Brownout detector was triggered
rst:0x3 (SW_RESET),boot:0x17 ...
```

**ESP32 内部欠压检测器（BOD，阈值 ~2.43V）在 12V 继电器吸合瞬间触发** —— CPU 被硬件复位，永远走不到 telemetry 阶段。这一条解释了全部表象：
- 用户看到的"演示模式" = 每次复位后开机前几秒的 OLED 初值
- "no connection" = 复位后还没二次连上 WiFi 就被用户拍照
- Dashboard 永远没数据 = 根本没机会调 send_state

#### 第 3 层：禁用 BOD 后浮现的 WiFi 状态机问题

写 `RTC_CNTL_BROWN_OUT_REG (0x3FF480D4)` 清 bit27/bit31 关闭 BOD 后：
- ✅ 12V COB 灯条跑完 60s 全程，CPU 没复位
- ✅ 12V 蠕动泵脉冲化后（1.5s × N + 1s gap）也不复位
- ❌ 但 `_send_telemetry` 仍然不输出任何 `[Telemetry] sent` —— 在 `if not wifi_client.is_connected(): return False` 处静默早返回
- ❌ 加诊断 print 后：`is_connected=False diag: (2, True, '0.0.0.0', False)` —— STA active=True 但 **ifconfig 显示 0.0.0.0**

也就是说：**外设全部初始化完成（WS2812 init、OLED I2C init、actuators 上电）之后，WiFi 在 ESP32 driver 层悄无声息地丢了 IP**，但 `network.WLAN(STA_IF).active()` 仍报 True。无任何 disconnect log，是 ESP32 idf-wifi 已知的"沉默掉线"模式。

启动日志能看到 `[WiFi] Connected, IP: 192.168.125.113` 一直保留到 init_system 结束；但进入主循环第一次决策周期（约 60s 后）时已是 `IP=0.0.0.0`。

#### 第 4 层：reconnect 也不工作

掉线后调 `wlan.connect()` 会抛 `Wifi Internal State Error`，因为 STA 卡在 `STA_CONNECTING`。即便先 `active(False)→active(True)→connect()`，12 秒、20 秒均 timeout。**唯一确认有效的恢复手段是 `machine.reset()`**（启动早期 RAM 干净时第一次 connect 总是成功）。

### 未解决 / 推迟

1. **WiFi 沉默掉线根因不明**：怀疑是 WS2812 持续刷新（GPIO26）或 OLED I2C 高频通信对 ESP32 idf-wifi 协议栈的影响，或路由器侧 idle 踢出。需要拆解：临时禁用 WS2812 + OLED 后观察 WiFi 是否能维持 5+ 分钟。
2. **AI 代理 `[Errno 118] EHOSTUNREACH / -203`**：与 telemetry 不在同一 IP，DNS/路由问题待查。
3. **DHT11 偶发 `attempt 1/3: [Errno 116] ETIMEDOUT`**：库内已重试 3 次，影响小。
4. **NTP 不同步导致 Day=0**：`config.NTP_SYNC_ON_CONNECT = False`（比赛现场优先稳定性的旧决策）。`wifi_client._sync_ntp()` 已实现但被 gate 关掉，需要时改 True。
5. **乱码字节串**：水泵动作前后 UART 出现 `\x80` 字节流，定位是 OLED I2C 错误时数据缓冲落到 stdout。继电器干扰使 I2C 总线异常。

### 硬件改善建议（写给现场）

软件层已经做到极限，根因仍在电源/EMI。建议：

1. **12V 水泵 + 12V COB 灯独立电源**：不要与 ESP32 的 5V USB 共享。当前共电源是 brownout 的物理根因。
2. **ESP32 VIN/5V 加 1000 µF 电解电容 + 0.1 µF 陶瓷电容并联**：吸收继电器开合的瞬态压降。
3. **继电器线圈两端反接 1N4007 续流二极管**：抑制断开瞬间的反向电势尖峰，减少 EMI。
4. **WS2812 数据线串 470Ω 电阻、电源加 100µF 退耦**：降低 WS2812 对 WiFi 2.4G 频段的串扰。
5. **如有条件，给 ESP32 单独配 5V/2A 适配器**，水泵/灯走另一路 12V 独立适配器（共地）。

具备硬件改善前，软件兜底（BOD 禁用 + 脉冲 + machine.reset 复位）能让系统跑起来，但每隔 3-5 分钟会复位一次，OLED 会有 ~30 秒空白期，dashboard 数据呈现"间歇刷新"而不是"持续流"。

### 调试工具产物

- `_serial_monitor.py`（项目根目录）：pyserial 通过 DTR/RTS 触发 ESP32 EN 引脚硬复位，并流式打印 UART 输出。绕过 mpremote 的 raw REPL 占用问题（main.py 跑起来后 mpremote 无法进入），是观察长时间运行行为的关键工具。

## 现场调试 #34 — 电源/brownout 误判更正 + 代码回退

### 起因

用户指出关键事实：**#33 观察到"水泵 brownout"时，12V 电源根本没打开**——继电器吸合时
水泵电机（负载侧）没通电、从未真正运转。也就是说当时的 brownout 复位**不是水泵电机涌流**造成的，
"水泵掉压"这个根因属于在错误前提下的凭空假设。`#33 第2层` 那条 `BOD triggered` 日志，更可能源于
当时 USB 5V 与 12V→稳压器 5V **双路互灌**到 ESP32 同一引脚的供电冲突，而非水泵。

### 独立供电下逐一复测（12V 已开、水管入水、电源域分开）

| 假设 | 实测 | 结论 |
|------|------|------|
| 水泵继电器吸合 → brownout 复位 | 泵真转，全程**无复位**，仅 RSSI 短暂掉 ~10dB | ❌ 未复现，非根因 |
| WS2812 关中断干扰 WiFi | 禁灯条后 WiFi 仍在 ~60s 掉线 | ❌ 排除 |
| 重连时 OOM（需 ~130KB 连续堆） | 改"软重连"不重建驱动后仍 20s 超时 | ❌ 非卡点 |
| **运行中掉线后无法重关联** | 开机能连(~5s)，运行 ~60s 掉线，软/硬重连均 20s 超时 | ✅ 真实现象，排查中 |

最强线索：轻量监听脚本 `wifi_stability.py`（不发遥测）同一热点稳跑 135s+ 零掉线；完整 `main.py`
向云端 43.156.68.157 发遥测且一直 ETIMEDOUT（疑似热点无外网），约 60s 后掉线连不回。
诱因疑为"失败的遥测 POST"或"热点无外网"，待验证。

### 回退的代码（基于误判加的电源 hack 全部移除）

| 文件 | 回退内容 | 现状 |
|------|---------|------|
| `main.py` | 删除启动早期写 `RTC_CNTL_BROWN_OUT_REG` 禁用 BOD 的代码 | BOD 恢复默认开启（真掉压时干净复位+重连，胜过静默乱掉） |
| `actuators.py` | `run_water_pump()` 脉冲分段 → 恢复连续运行（1s 分段 sleep 便于中断） | 与 `run_light` 同结构 |
| `action_runtime.py` | 删除 water/light 后的 `time.sleep(1.5)`（"给 WiFi 恢复"） | — |
| `config.py` / `status_strip.py` | 删除诊断开关 `WS2812_ENABLED`（曾用于排查 WS2812 干扰） | 灯条恢复常开 |
| `main.py` | 删除 `_send_telemetry` 内临时 `gc.mem_free()` 调试 print | — |
| `wifi_stability.py` | `STRESS_ACTUATOR` 恢复默认 `False` | 不会误触发水泵 |

> 保留未动：`wifi_client.is_connected()` 基于 IP 判断、`connect()` 软重连优先 + 失败回退硬复位、
> `_wifi_fail_streak>=3 → machine.reset()` 兜底、loop 决策前先发遥测——这些是 WiFi 通路处理，
> 与电源误判无关，且 WiFi 重关联问题仍在排查中。

### 教训

**做硬件诊断前先确认实际供电/接线状态与你以为的一致。** 这次在 12V 没开的前提下推导出
"水泵涌流掉压"，并为此禁了 BOD、把水泵改成脉冲、到处加 sleep——全是治一个不存在的病。
代价是掩盖了真正的问题（运行中 WiFi 重关联失败），还引入了 BOD 关闭这种有害 hack。
排查工具 `tools/serial_monitor.py`（被动读串口）和 `wifi_stability.py`（在线监测 + RSSI）保留备用。
